Skills: How to Build Your Own and Why It Changes Everything

9 min read · Updated May 2026 · throughlouislens

What a skill is, why it exists, how it's structured, and how to create one from scratch

What Is a Markdown File?

Before anything else, a quick primer, because skills are written in markdown, and if you've never heard of it, it's worth knowing what it is.

Markdown is a plain text format for writing structured content. You write it in any text editor, and it renders cleanly as formatted content. No special software needed. A .md file is just a text file that uses simple symbols to indicate structure.

The basics:

# Heading 1
## Heading 2
### Heading 3

**bold text**
*italic text*

- bullet point
- another point

1. numbered item
2. second item

`inline code`

code block

That's 90% of what you'll use. Markdown is the standard format for documentation, README files, wikis, and for our purposes, skills. It's human-readable when raw, and it renders cleanly when displayed. If you can write a Word document, you can write markdown.

What Is a Skill?

A skill is a set of instructions that loads into Claude automatically when it's needed, without you having to re-explain anything.

Think of it this way: every time you open a new chat with Claude, it starts fresh. It doesn't remember your preferences, your project structure, your naming conventions, or how you like your documents formatted. Without skills, you'd have to re-explain all of that at the start of every session.

A skill solves that. It's a markdown file that says: "When this kind of task comes up, here's everything Claude needs to know to do it correctly, every time."

You install the skill once. After that, Claude reads it automatically when it's relevant. The rules apply without you having to repeat them.

Why Build a Skill?

The short answer: because you'll repeat yourself otherwise.

The longer answer is that there are certain types of knowledge that belong in a skill rather than in a chat.

Formatting standards. If your workspace always uses specific icons, Title Case headings, and a fixed colour system for statuses, that's a skill. You describe it once and every page Claude touches will follow those rules.

Workspace structure. If your project management tool has a specific folder hierarchy, a naming convention, and a set of IDs that Claude needs to know to put things in the right place, that's a skill. Without it, Claude has to figure out the structure from scratch every session, which is slow and error-prone.

Document templates. If every report you produce needs a specific layout, a specific section order, and a specific table format, that's a skill. Write it once and every document Claude produces follows the template.

Brand guidelines. If your brand has specific hex codes, typefaces, and tone of voice rules, that's a skill. Attach it once and every piece of content Claude helps create stays on-brand.

Domain knowledge. If you work in a specialised field and Claude keeps needing background context to be useful, that's a skill. Load the background once rather than pasting it into every conversation.

The rule of thumb: if you've typed the same instruction more than twice, it probably belongs in a skill.

What Goes Inside a Skill?

A skill has two components.

1. SKILL.md, the main instruction file. This is what Claude reads. It contains:

  • Frontmatter (name, description)
  • A role definition (what this skill is for)
  • Trigger conditions (when should this skill activate)
  • Rules and behaviour (the actual instructions)
  • Structure and format notes (if the skill produces output, how should it look)
  • References to any asset files

2. An assets/ folder (optional), for supporting reference files. Any data, tables, or reference content that's too long to live inside SKILL.md goes here as separate files. SKILL.md points to them and never duplicates the content.

That's it. One folder, one or two files, total clarity.

The Structure of a Skill

Here is the exact directory layout:

your-skill-name/
├── SKILL.md
└── assets/
    └── reference-data.md

If you have no assets, skip the folder entirely:

your-skill-name/
└── SKILL.md

The SKILL.md File

Every SKILL.md starts with YAML frontmatter, the block between the --- lines. This is non-negotiable. Without it, the packager rejects the file.

---
name: skill-name
description: "One sentence describing when this skill should trigger and what it does."
---

The description field is the most important thing you'll write. It's what determines whether Claude picks up the skill automatically. Write it as a clear, specific trigger condition.

Bad description: "Helps with formatting."

Good description: "Use this skill whenever creating, editing, or building any Notion page or database. Enforces consistent formatting rules: SVG icons only, Title Case headings, and standardised layout conventions."

After the frontmatter, the body of SKILL.md follows this structure:

# SKILL: Your Skill Title

**Owner:** [Your name or team]
**Scope:** [What this skill applies to]

-----

## ROLE DEFINITION
What this skill is for and who it serves. One short paragraph.

-----

## WHEN THIS SKILL IS TRIGGERED
- Specific condition 1
- Specific condition 2
- Specific condition 3

-----

## RULES / BEHAVIOUR
1. The actual rules Claude must follow. Be specific.
2. Not principles — actual rules. "Always use Title Case" not "aim for consistency".
3. If the rule has an exception, write the exception explicitly.

-----

## STRUCTURE / FORMAT
If this skill produces output (a document, a database, a report), describe the exact format here.
If the detail is long, reference an asset file instead.

-----

## REFERENCE / ASSETS
See assets/reference-data.md

-----

*Skill version: 1.0*
*Last updated: [date]*
*Change: [what changed from the previous version]*

Asset Files

Asset files live in the assets/ folder. They follow a simple format:

# Asset Title

Tables, lists, reference data, or detailed specifications
that are too long to sit inside SKILL.md.

The key rule: SKILL.md points to assets. Assets never contain the same content as SKILL.md. Each file holds one thing, kept completely separate so either can be updated without touching the other.

Real Examples of Skills

To make this concrete, here are three types of skills and what they actually contain.

A formatting skill (e.g., Notion Format) The description triggers on any Notion page or database creation. The body defines icon rules (SVG only, no emoji), heading conventions (Title Case throughout), a status colour system (grey for inactive, blue for active, green for complete), and property ordering rules. An asset file holds the full icon library and colour mapping table. Result: every Notion page Claude touches looks identical, no matter when you're working.

A workspace structure skill (e.g., ClickUp Structure) The description triggers whenever creating tasks, lists, or folders in the workspace. The body defines the full Space-Folder-List hierarchy with IDs, naming conventions (ALL CAPS for folders, Word_Word format for lists), and rules for where things go. An asset file holds the full ID map. Result: Claude always puts things in exactly the right place without having to look it up.

A document template skill (e.g., Brand Guidelines) The description triggers whenever producing a document or piece of content that should match the brand. The body defines hex codes for the colour palette, typeface names and weights, tone of voice rules, and what the brand never does. No asset needed if everything fits in SKILL.md. Result: every document produced stays on-brand automatically.

How to Write a Description That Actually Triggers

The description field is Claude's cue. If it's vague, the skill won't load when it should. If it's too broad, it'll load when it shouldn't. Getting this right is the difference between a skill that works invisibly and one you have to invoke manually every time.

Write the description as a sentence that answers: "When should this load, and what does it do?"

Template: "Use this skill whenever [specific trigger]. [What it does, one clause]."

Examples:

  • "Use this skill whenever creating or formatting any Word document. Applies the company letterhead, heading styles, and table formatting conventions."
  • "Use this skill whenever working in the project management workspace. Contains the full folder hierarchy, list IDs, and naming conventions."
  • "Use this skill whenever producing content for the brand. Enforces colour palette, typeface choices, and tone of voice rules."

Avoid vague words like "relevant", "appropriate", or "when needed". Be specific about the action (creating, editing, building, formatting) and the context (Notion, ClickUp, a Word document, brand content).

The Seven Rules for Building a Good Skill

  1. Frontmatter is required. No --- block, no working skill.
  2. SKILL.md lives at the root of the folder, always.
  3. Assets go in assets/, never embedded inside SKILL.md.
  4. Write the description like a trigger condition, not a summary.
  5. Use rules, not principles. "Always" and "never" are your friends.
  6. Bump the version number every time you update the skill.
  7. Log what changed in the *Change:* line, future you will thank you.

How to Get Claude to Build a Skill for You

If you know what you want but you're not sure how to write it, give Claude the raw information and ask it to structure the skill for you. The prompt pattern that works:

"I want to create a skill for [specific task or context]. Here's what it needs to know: [paste your rules, your data, your formatting requirements, your reference tables, all of it]. Structure this as a SKILL.md file with the correct frontmatter, and create any asset files if the content is too long to fit cleanly in one file."

The more specific your input, the better the skill. Don't say "it should know my formatting preferences", say "headings are always Title Case, icons are SVG only, status colours follow this system: grey for inactive, blue for active, green for complete." Raw, specific detail produces a skill that actually works.

If you have an existing document, spreadsheet, or set of notes that captures the knowledge you want in the skill, paste it in or attach it. Claude can read it and extract the rules, conventions, and data into the correct skill format.

A Blank Skill Template to Copy

Use this as your starting point for any new skill:

---
name: your-skill-name
description: "Use this skill whenever [specific trigger]. [What it does, one clause]."
---

# SKILL: Your Skill Title

**Owner:** [Your name]
**Scope:** [What this applies to]

-----

## ROLE DEFINITION
One paragraph: what this skill is for and who uses it.

-----

## WHEN THIS SKILL IS TRIGGERED
- [Specific condition 1]
- [Specific condition 2]
- [Specific condition 3]

-----

## RULES / BEHAVIOUR
1. [Rule one — specific and unambiguous]
2. [Rule two]
3. [Rule three]

-----

## STRUCTURE / FORMAT
[If this skill produces output, describe the exact format here.
If you have a lot of detail, create an asset file and reference it below.]

-----

## REFERENCE / ASSETS
See assets/[your-asset-file].md

-----

*Skill version: 1.0*
*Last updated: [date]*
*Change: Initial version*

Fill in the blanks, save it as SKILL.md inside a folder named after your skill, and you're done.
